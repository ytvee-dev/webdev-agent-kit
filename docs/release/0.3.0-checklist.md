# WebDev Agent Kit 0.3.0 Release Checklist

Planned tag: `v0.3.0`. Preparing this checklist does not authorize merging the
release pull request, creating the tag, or publishing a GitHub Release.

## Before Merge

- Confirm the release pull request contains every planned milestone and no
  unrelated source changes.
- Require successful `skill-pack-ci`, `quality-ci`, and `link-check` results on
  the final head commit.
- Review `CHANGELOG.md` Unreleased entries against the actual pull request.
- Review user-facing installation claims against `bundle-manifest.json`, built
  targets, extracted archives, and official client documentation.
- Confirm existing host `AGENTS.md`, `CLAUDE.md`, and README files are not
  silently created, replaced, or edited by runtime archives.

## Local Release Candidate

Run from a clean source checkout:

```text
python scripts/validate_skill_pack.py
python scripts/validate_release_tag.py --tag v0.3.0
python scripts/validate_install_guides.py
python scripts/build_skill_targets.py
python scripts/build_release_archives.py --output release-artifacts --version v0.3.0
python scripts/validate_release_archive.py --directory release-artifacts --version v0.3.0
python scripts/build_release_notes.py --tag v0.3.0 --output release-artifacts/RELEASE_NOTES.md
```

Build the archives twice in separate empty directories and compare SHA-256
digests. Deterministic builds must produce identical hashes.

## Manual Artifact Inspection

- Codex and VS Code Codex archives start at `.agents/` and include
  `.agents/.codex-plugin/plugin.json` plus `.agents/skills/`.
- Cursor starts at `.agents/` and root `.cursor/`, with no nested
  `.agents/.cursor/`.
- Claude Code and VS Code Claude start at `webdev-agent-kit/`, include
  `.claude-plugin/plugin.json` and `skills/`, and contain no Codex metadata.
- Every stable archive is byte-identical to its versioned counterpart.
- `SHA256SUMS` contains every stable and versioned archive exactly once.
- No archive contains human-facing docs, `project/`, `examples/`, dependencies,
  caches, symlinks, unsafe paths, or root project instructions.

## Publish Gate

Only after explicit release approval and merge:

1. Create the exact annotated or lightweight tag `v0.3.0` from the approved
   release commit.
2. Push that tag once. The tag-triggered workflow validates source, targets,
   archives, install guides, release notes, and tag/version equality before
   publishing.
3. Confirm all ten stable/versioned archives and `SHA256SUMS` are attached to
   the GitHub Release.
4. Download one archive per canonical target, verify its checksum, extract it
   into an empty fixture, and confirm the documented entrypoint.

Do not repoint the tag or overwrite release assets unless a separately approved
recovery plan explains the incident and replacement.
